Good morning everyone.

Today I'd like to walk you through the evolution of our Isolated Recovery Environment architecture on AWS.

Rather than jumping straight to the final solution, I'd like to show the design journey we went through, the alternatives we evaluated, the trade-offs we identified, and why we finally arrived at the recommended three-VPC architecture.

(Pause)

The objective throughout this exercise was very simple:

Design an environment where we can recover securely even if production itself can no longer be trusted.

--------------
Slide 2 – Journey

This slide summarizes our design evolution.

We evaluated four different approaches.

We started with a traditional subnet-based isolation model because it's simple and easy to build.

As we analyzed the security risks, we explored a highly segmented five-VPC design.

Although it significantly improved isolation, it introduced a lot of operational complexity.

To reduce that complexity, we then looked at a two-VPC approach.

That simplified the design, but it also combined administrative access with the recovery environment, which weakened one of our core security principles.

Finally, we arrived at the three-VPC architecture, which gives us the right balance between security, operational simplicity, and automation.
-----------------
Slide 3 – Phase 1

Let me start with the original design.

Everything existed inside a single VPC.

We achieved separation only through subnets, route tables, and security groups.

From an infrastructure perspective, this is straightforward and easy to manage.

However, all workloads still belong to the same trust boundary.

If an attacker compromises privileged credentials or Active Directory, lateral movement becomes much easier because the entire recovery environment shares the same network boundary.

So while subnet isolation provides logical separation, it doesn't provide the level of isolation we're looking for in a cyber recovery environment.

-----------------
Slide 4

These are the primary limitations we identified.

Administrative traffic, workloads, backups, and recovery infrastructure all coexist within the same VPC.

That increases the blast radius if administrative credentials are compromised.

It also makes it difficult to demonstrate true isolation from a compliance perspective.

That led us to ask whether VPC-level isolation would give us stronger security boundaries.

-----------------
Slide 5 – Five VPC

Our first attempt was to maximize isolation.

We separated every major function into its own VPC.

Administrative access, analytics, vault, forensics, and shared services all became independent trust boundaries.

From a security standpoint, this approach was excellent.
-----------------
Slide 6

However, once we started looking at the operational side, we realized the design had become overly complex.

Every VPC required Transit Gateway attachments, routing, endpoints, inspection policies, and ongoing maintenance.

Recovery orchestration also became more complicated because workflows had to cross multiple trust boundaries.

In other words, we had solved one problem, but introduced another.

So the next question became:

Can we simplify the architecture without giving up too much security?
-----------------
Slide 7 – Two VPC

That's where the two-VPC proposal came in.

This was intended to reduce operational complexity while preserving the protected vault.

One VPC would contain both administrative access and the recovery platform.

The second VPC would contain the immutable recovery repository.
-------------------
Slide 8

Operationally, this was much easier to manage.

But from a security perspective, there was one concern.

The administrative entry point now shares the same trust boundary as our automation platform.

If that entry point were ever compromised, the attacker is already inside the environment that controls recovery automation.

Although the protected vault remains isolated, we lose separation between people accessing the environment and the systems performing recovery.

That didn't align with our Zero Trust objectives.
-----------------------
Slide 9 – Three VPC

Smile here.

This brings us to the final architecture.

Instead of maximizing isolation or minimizing complexity, we focused on balancing both.

We separated the environment into three independent trust boundaries.

Each VPC has a single responsibility.

Administrative access.

Recovery execution.

Protected recovery assets.
------------------------
Slide 10 – Three VPC Explanation

Recovery Access VPC exists only for administrators.

Core Recovery VPC exists only for automation and recovered workloads.

Protected Data VPC exists only for immutable recovery assets.

None of these VPCs perform each other's responsibilities.

That separation significantly reduces the blast radius while keeping the architecture operationally manageable.
----------------------------
Slide 11 – Final Architecture
Now I'd like to walk through the final architecture end-to-end.

Start left.

Everything begins with the recovery administrator.

During a cyber event, administrators authenticate using our enterprise identity provider with MFA enabled.

Point.

They establish connectivity through AWS Client VPN into the Recovery Access VPC.

This VPC exists purely for administrative access.

It does not host recovered applications or recovery data.

Move slightly right.

From there, administrators access the recovery environment using Systems Manager Session Manager.

This eliminates the need for public SSH or RDP access.

Every administrative action is authenticated, authorized, and logged.


Move to Core Recovery.

Once inside, we reach the Core Recovery VPC.
This is where the actual recovery activities take place.
Terraform provisions the required infrastructure.
Red Hat Ansible Automation Platform performs operating system configuration, middleware deployment, application deployment, and post-recovery validation.
AWS Managed Microsoft Active Directory provides isolated identity services for recovered workloads.

Move to TGW.
Communication between VPCs happens through Transit Gateway.
Rather than allowing unrestricted communication, traffic is directed through our centralized inspection layer.
Point to firewall.
AWS Network Firewall operating in appliance mode performs stateful inspection before traffic reaches another trust boundary.
This ensures every connection between VPCs follows our Zero Trust networking principles.

Move to Protected Data.
Finally, we reach the Protected Data VPC.

This contains our recovery repository.
Immutable S3 Object Lock backups.
Metadata stored in DynamoDB.
File repositories.
Landing zones.
Malware scanning.

Validation before restoration.

Recovery assets remain isolated until they are requested through approved automation.

Once recovery validation is complete and business owners approve the recovered workloads, applications can be promoted for production use.

---------------------------------

Last Slides

The remaining slides summarize the architecture.

One of the principles we followed throughout the design was keeping every VPC focused on a single responsibility.

Recovery Access handles administrators.

Core Recovery handles recovery execution.

Protected Data safeguards trusted recovery assets.

That separation gives us strong security boundaries while still keeping the solution practical to operate.

Final Closing
--------------
To summarize, our goal wasn't simply to design another disaster recovery environment.

Our objective was to build a recovery platform that we can trust even when production has been compromised.

We evaluated multiple architectures before arriving at the final design.

The recommended three-VPC architecture provides the right balance of isolation, operational simplicity, automation, and governance.

It aligns with Zero Trust principles, supports secure cyber recovery, and gives us a scalable foundation that we can continue to build on as additional recovery capabilities are introduced.
